# 🏗 Архитектура решения "SecureField MVP" (Android-First)

Мы строим монолитное Android-приложение с чистой архитектурой (Clean Architecture), готовое к миграции на KMP в будущем (бизнес-логика будет изолирована от Android-зависимостей).

## 1. Стек технологий

*   **Язык:** Kotlin 1.9+
*   **UI:** Jetpack Compose (Material 3)
*   **DI:** Koin (легче для KMP, чем Hilt)
*   **Async:** Kotlin Coroutines + Flow
*   **ML Runtime:** ONNX Runtime (Mobile)
*   **OCR:** PaddleOCR v4/v5 (Quantized INT8 models)
*   **PDF Engine:** **iText 7 Core** (Community/AGPL для старта, либо PDFBox-Android, если лицензия критична). *Примечание: iText имеет строгую AGPL, если код закрыт — нужно покупать. PDFBox — Apache 2.0 (бесплатно), но сложнее.* **Рекомендую начать с PDFBox-Android для MVP, чтобы не платить сразу.**
*   **Database:** Room + SQLCipher (net.zetetic:android-database-sqlcipher)
*   **Matching:** `me.xdrop:fuzzywuzzy` (Java порт) или нативная реализация Levenshtein.

## 3. Детальный план реализации (8 недель)

### Фаза 1: Фундамент и OCR (Недели 1-2)

**Цель:** Приложение открывает камеру, находит текст, создает PDF.

1.  **Настройка проекта:**
    *   Интеграция `CameraX` (режим `ImageAnalysis` для превью и `ImageCapture` для фото).
    *   Подключение `ONNX Runtime` и загрузка моделей PaddleOCR (`det` и `rec`) из assets.
2.  **ML Pipeline:**
    *   Реализовать `OcrEngine`: Bitmap -> Resize -> Normalize -> ONNX Inference -> Coordinates & Text.
    *   Важно: Сохранять не только текст, но и `BoundingBox` для каждого слова.
3.  **PDF Generation:**
    *   Использовать `PdfDocument` (нативный) для создания нового PDF из фото.
    *   Нарисовать невидимый текст поверх фото (Text Paint c `alpha = 0` или режим рендеринга `INVISIBLE`).

**Результат:** Приложение "SecureScanner", которое делает Searchable PDF оффлайн.

### Фаза 2: Smart Redaction (Недели 3-4)

**Цель:** Реализовать замазывание данных (визуально + деструктивно).

1.  **Логика поиска (Matcher):**
    *   Написать Regex-паттерны для кредиток (Luhn algorithm), дат, Email, Passport MRZ.
    *   Реализовать `TextFinder`: ищет пересечение OCR-боксов с найденными паттернами.
2.  **UI Редактора:**
    *   Отображение фото с нарисованными прямоугольниками (Canvas).
    *   Жесты: Тап по слову -> Скрыть/Показать. Рисование пальцем (Custom Mask).
3.  **Деструктивное сохранение (PDFBox/iText):**
    *   Загрузка созданного PDF.
    *   `PDPageContentStream`: Рисуем черный прямоугольник.
    *   **Критично:** Удаление текстовых операторов (TJ/Tj) под прямоугольником. В PDFBox это сложно (нужно парсить контент-стрим), в iText проще (`PdfCleanUp`).
    *   *Совет:* Для MVP просто рисуем черный квадрат и *не добавляем* невидимый текст в эту область при генерации. Это проще, чем удалять.

**Результат:** Приложение умеет скрывать данные и экспортировать "безопасный" PDF.

### Фаза 3: Data Matching & Storage (Недели 5-6)

**Цель:** Импорт Excel и сверка.

1.  **Secure Storage:**
    *   Настройка Room + SQLCipher.
    *   `Entity`: `ProductItem (sku, name, vector_embedding?)`.
2.  **Import:**
    *   Парсер CSV (OpenCSV).
    *   Маппер полей: Пользователь выбирает, какая колонка - "Артикул", какая - "Имя".
3.  **Matching Engine:**
    *   Реализовать `FuzzyMatcher`: Сравнивает OCR-текст с БД.
    *   Оптимизация: Сначала точное совпадение (HashMap), потом Fuzzy (если записей < 5000). Если больше — нужен FTS (Full Text Search) в SQLite.
4.  **AR Overlay:**
    *   В режиме камеры рисовать рамки поверх текста: Зеленая (Найдено), Красная (Нет в базе). *Внимание: Рамки будут "дрожать", для MVP это ок. Стабилизация — задача на будущее.*

**Результат:** Приложение "видит" знакомые артикулы.

### Фаза 4: Лицензирование и Полировка (Недели 7-8)

**Цель:** Защита самого приложения и подготовка к продаже.

1.  **Security Hardening:**
    *   Добавить `FLAG_SECURE`.
    *   Проверка подписи APK (чтобы не взломали и не вырезали проверку лицензии).
2.  **Licensing Manager:**
    *   Генерация `DeviceID`.
    *   Экран ввода лицензии -> Проверка подписи RSA.
    *   Блокировка функций без лицензии (или водяной знак).
3.  **UI/UX Polish:**
    *   Онбординг экраны.
    *   Анимации загрузки (OCR может занять 1-2 сек).
    *   Обработка ошибок ("Файл поврежден", "Камера недоступна").

## 4. Обоснование технических решений

1.  **Почему PDFBox, а не iText?**
    *   iText 7 требует покупать лицензию (~$1000+) или открывать исходный код (AGPL). Для стартапа это риск.
    *   PDFBox (Apache 2.0) бесплатен. Он тяжелее и медленнее, но для MVP (генерация простых документов) его хватит. Если пойдут продажи — купите iText.

2.  **Почему SQLCipher?**
    *   Это стандарт индустрии. Если вы продаете "Privacy", шифрование базы "at rest" (в покое) — обязательное требование Enterprise-клиентов.

3.  **Почему PaddleOCR INT8?**
    *   FP32 (полная точность) весит ~15MB, но INT8 весит ~5MB и работает на 30-50% быстрее на CPU. Потеря точности минимальна для печатного текста.

## 5. План тестирования и Валидации

1.  **Security Audit (Self-Check):**
    *   Сгенерировать PDF с Redaction.
    *   Открыть в Adobe Acrobat Pro -> Tools -> Edit PDF.
    *   Попробовать удалить черный квадрат.
    *   *Успех:* Под квадратом пустота или белый фон. Текста нет в поиске (Ctrl+F).

2.  **Performance Test:**
    *   Устройство: Бюджетный Android (Snapdragon 680, 4GB RAM).
    *   Сценарий: Фото листа А4 с мелким текстом.
    *   *Критерий:* OCR < 3 сек, UI не фризит.

3.  **Matching Test:**
    *   База: 10,000 артикулов.
    *   Поиск: Нечеткий поиск одного артикула.
    *   *Критерий:* Результат < 200мс. (Если больше — нужно внедрять FTS4/FTS5 в SQLite).
